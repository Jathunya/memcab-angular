## A. Test Plan Summary

### A.1 Scope of Testing

**In scope:**

| Area | Coverage |
|:--|:--|
| Authentication (Firebase) | Login, signup, logout, password reset, account deletion, error-message mapping |
| Per-account data isolation | localStorage scoping by `uid` (regression coverage for the cross-account deck leak fix) |
| Signup validation | Real-time password length feedback, confirm-password match, submit gating |
| Daily Review (SRS) | Confidence-based mastery state machine (New → Learning → Mastered), flashcard flip, TTS |
| Responsive shell | Sidebar ↔ bottom-nav breakpoint, mobile avatar routing, Settings horizontal tab scroll |
| Forgot Password UI | Inline card state swap, validation messaging, success/error banners |
| Sentence Lab — server-side Groq proxy | Client calls `/api/generate-sentence` (a Vercel serverless function), never Groq directly; the key-panel UI is gone entirely; loading/error/success states; save-to-deck wiring |

**Out of scope (explicitly, with reason):**

| Area | Why excluded |
|:--|:--|
| Spaced-repetition interval scheduling (SM-2/Easiness Factor) | **Not implemented.** The app's SRS is a 3-state Leitner-style confidence model (`reviewWord()` in `store.service.ts`), not an interval/EF scheduler. Test cases below cover the state machine that actually exists — do not write test cases against a formula that isn't in the code. |
| Sentence Lab's LLM output correctness | Output is generated by a third-party model (Groq); grading sentence *quality* is a product/content review task, not a QA automation target. Correctness of the proxy plumbing (request/response contract, error surfacing) is in scope; what the model says is not. |
| Real, live Firebase project calls in CI | See §A.3 — mocked at the network layer for determinism; live-project smoke testing is a separate, periodic manual/scheduled activity. |
| Real, live Groq calls in CI | Same rationale as above, see §A.3b — `/api/generate-sentence` is mocked at the network layer in the checked-in suite. |

### A.3 Testing Strategy Note: Mocking vs. Live Firebase

E2E specs intercept Firebase's Identity Toolkit REST calls rather than hitting the live `memcab-19762` project. This is a deliberate flakiness-reduction decision:

- **Pro:** Deterministic, offline-capable, no rate limits, no leftover test accounts in a real user pool, safe for CI.
- **Con:** The mock fixtures reproduce Identity Toolkit's *observed* wire format for `firebase@^12.16.0`. They are not a documented-stable contract. A future major SDK upgrade could silently invalidate them.
- **Mitigation:** Pair this suite with a periodic (e.g. weekly) manual or scheduled smoke run against a real, non-production Firebase project to catch drift the mocks wouldn't. This is a recommendation for the team's CI setup, not something this suite does automatically.

### A.3b Testing Strategy Note: Mocking vs. Live Groq (Sentence Lab)

Same rationale and same shape of tradeoff as §A.3, applied to `api/generate-sentence.ts`:

- The checked-in suite (`e2e/sentence-lab.spec.ts`) mocks `/api/generate-sentence` directly — it never calls the real serverless function or Groq, so it needs no `GROQ_API_KEY` and costs nothing to run repeatedly in CI.
- Before this suite was written, the real end-to-end path was verified once, manually: `vercel dev` running locally with a real key in `.env`, driving the actual UI through a real Firebase signup, a real Groq-generated sentence, and the save-to-deck flow. That pass is not automated or repeated by CI — it's a one-time confirmation that the architecture (browser → `/api/generate-sentence` → Groq, key never in the client bundle) actually works, not a regression guard.
- **Recommendation:** if `api/generate-sentence.ts` or the Groq prompt/model changes materially, repeat that manual `vercel dev` pass before shipping — the mocked E2E spec will happily stay green even if the real Groq integration breaks, since it never touches Groq at all.

### A.4 Exit Criteria

Regression testing before a release is considered complete when:

1. `npm test` — 100% pass, zero skipped tests without a linked tracking ticket.
2. `npm run e2e` — 100% pass on `chromium-desktop`.
3. `ng build` — zero errors, zero new warnings vs. the previous release build.
4. All **Critical** and **High** severity bugs opened during the cycle are either fixed-and-verified or explicitly deferred with sign-off (not silently left open).
5. Any new user-facing flow shipped in the release has at least one corresponding automated test (unit or E2E) — no untested happy path merges.

## D. Manual Test Checklist (Pre-Deployment Sign-off)

Run this by hand, once, after setting `GROQ_API_KEY` in the Vercel Dashboard and redeploying — before telling anyone else the release is ready. It's a human sanity pass, not a replacement for §A.4's automated exit criteria.

### D.1 End-to-End Verification — Sentence Lab in Production

1. Confirm the deploy actually picked up the new env var: Vercel Dashboard → your project → **Deployments** → latest → **Source** should be *after* you added `GROQ_API_KEY`. If the latest deployment predates it, redeploy first — existing deployments don't retroactively pick up new env vars (see the note at the end of the earlier conversation on this).
2. Open the live production URL in a **fresh/incognito** window (avoids stale cached JS or leftover localStorage from local dev).
3. Sign up or log in with a real (or disposable) account.
4. Navigate to **Sentence Lab** from the sidebar.
5. Confirm there is **no** "Add your Groq API key" panel anywhere on the page — if you see one, the deploy shipped the old client bundle.
6. Type a real word (e.g. "generous") and click **✨ AI Generate**.
7. Confirm, in order: the spinner ("Composing your sentence…") appears → then a result card appears with a Thai word, an English example sentence, a part-of-speech tag, and (usually) a pronunciation/IPA line.
8. Click both 🔊 speaker icons (next to the word and the sentence) and confirm audio actually plays.
9. Click **💾 Add to My Decks**, pick a deck (or create one), save, and confirm the word shows up in that deck under **My Decks**.
10. Repeat steps 6–9 once more with a second, different word — confirms this isn't a one-shot fluke.

### D.2 UI/UX Consistency Check

Memcab's actual design language is the **warm cream / neumorphic soft-UI** system (see `styles.css` design tokens), not a "cinematic" look — check against that:

- [ ] Light mode: background is warm cream (`#f7f3ec`), not stark white; cards have the soft dual-shadow "raised" look, not flat borders.
- [ ] Toggle dark mode in Settings → App Preferences: background flips to charcoal slate (`#0f172a`), text stays legible everywhere (this is the exact contrast bug that was fixed on the Daily Review card — re-check that card specifically in dark mode).
- [ ] Thai script renders correctly (no tofu/missing-glyph boxes) in the Sentence Lab result, the flashcard back face, and the sidebar/topbar Thai greeting.
- [ ] Resize the browser across the 768px breakpoint: sidebar ↔ bottom nav swap cleanly, nothing overlaps or double-renders during the transition.
- [ ] Sentence Lab's result card, buttons, and spacing look consistent with the rest of the app (same card radius, same shadow depth, same teal accent) — nothing looks like it was bolted on.
- [ ] Hover/press states on buttons show the expected subtle scale/lift (not a dead, static click).

### D.3 Error Handling — Reading the Console

1. Open DevTools (F12) → **Console** tab *before* triggering anything, so you don't miss early errors.
2. Trigger a Sentence Lab generation and watch for:
   - `[SentenceService] generateCustom failed: ...` — logged client-side on any failure; the *cause* is in this log even though the UI only shows the generic "Could not generate sentence" message to the user.
   - Any **uncaught** red error (not a `console.error` from the app's own logging) — this indicates a real bug, not an expected/handled failure.
3. Open the **Network** tab, filter by `generate-sentence`, and inspect the request to `/api/generate-sentence`:
   - Status `200` = success.
   - Status `500` with body `"Sentence generation is not configured on the server."` = `GROQ_API_KEY` is missing/empty in Vercel's env vars for this environment — go back to D.1 step 1.
   - Status `502` = Groq itself rejected or failed the request (bad/revoked key, rate limit, Groq outage) — check Vercel's **Functions** log tab for the real Groq error, which is logged server-side (`console.error('[api/generate-sentence] Groq request failed...')`) but never sent to the client.
4. Also check the **Functions** tab in the Vercel Dashboard (Project → Deployments → latest → Functions → `generate-sentence`) for server-side error logs — the browser console only ever sees the friendly, generic message by design (see `OFFLINE_ERROR` in `sentence.service.ts`); the real cause lives in the function logs.

### D.4 Cleanup Check — No Legacy Artifacts Interfering

The old client-side "bring your own key" flow stored data in `localStorage` that the app now actively purges on load — confirm the purge actually worked rather than assuming it:

1. DevTools → **Application** tab → **Local Storage** → your production origin.
2. Confirm these keys are **absent** (the app removes them itself on `SentenceService` construction — see `LEGACY_KEY_STORAGE_KEYS` in `sentence.service.ts`):
   - `MEMCAB_GROQ_KEY`
   - `MEMCAB_GEMINI_KEY`
   - `MEMCAB_OPENAI_KEY`
3. Separately, confirm the **unscoped, pre-data-isolation-fix** keys are also absent (these predate the per-account `uid` scoping fix and should never reappear):
   - `memcab_folders` (no `::uid` suffix)
   - `memcab_progress` (no `::uid` suffix)
   - If either shows up, decks could leak across accounts again — treat this as a **Critical** bug per TC-005, not a cosmetic leftover.
4. Confirm the keys that *should* be present instead follow the `::<uid>` pattern, e.g. `memcab_folders::AbCdEf123...` — open one and spot-check it's valid JSON containing only decks that belong to the account you're currently signed in as.
