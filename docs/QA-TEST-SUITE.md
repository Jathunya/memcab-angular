## A. Test Plan Summary

### A.1 Scope of Testing

**In scope:**

| Area | Coverage |
|:--|:--|
| Authentication (Firebase) | Login, signup, logout, password reset, account deletion, error-message mapping |
| Per-account data isolation | localStorage scoping by `uid` (regression coverage for the cross-account deck leak fix) |
| Signup validation | Real-time password length feedback, confirm-password match, submit gating |
| Daily Review (SRS) | Confidence-based mastery state machine (New → Learning → Mastered), flashcard flip, TTS |
| Responsive shell | Sidebar ↔ bottom-nav breakpoint, mobile avatar routing, Settings horizontal tab scroll |
| Forgot Password UI | Inline card state swap, validation messaging, success/error banners |

**Out of scope (explicitly, with reason):**

| Area | Why excluded |
|:--|:--|
| Spaced-repetition interval scheduling (SM-2/Easiness Factor) | **Not implemented.** The app's SRS is a 3-state Leitner-style confidence model (`reviewWord()` in `store.service.ts`), not an interval/EF scheduler. Test cases below cover the state machine that actually exists — do not write test cases against a formula that isn't in the code. |
| Sentence Lab's LLM output correctness | Output is generated by a third-party model (Groq); grading sentence *quality* is a product/content review task, not a QA automation target. UI wiring (loading state, error handling, save-to-deck) is in scope. |
| Real, live Firebase project calls in CI | See §A.3 — mocked at the network layer for determinism; live-project smoke testing is a separate, periodic manual/scheduled activity. |

### A.2 Environment Setup

| Layer | Tool / Version |
|:--|:--|
| Framework under test | Angular 21.2, zoneless, Signals-driven state |
| Unit test runner | Vitest, via `@angular/build:unit-test` (`ng test`) |
| E2E runner | Playwright (`@playwright/test`), Chromium |
| Local dev server | `ng serve`, auto-started by `playwright.config.ts` on port 4310 for E2E |
| Network mocking (E2E) | `page.route()` interception of Identity Toolkit endpoints — see `e2e/mocks/firebase-auth-mock.ts` |
| Mobile viewport reference | iPhone 14 Pro, 393×851 |

**Running the suite:**

```bash
npm test          # unit tests (Vitest)
npm run e2e        # E2E tests (Playwright, headless)
npm run e2e:ui      # E2E tests, interactive UI mode (debugging)
npm run e2e:report   # Open the last HTML report
```

### A.3 Testing Strategy Note: Mocking vs. Live Firebase

E2E specs intercept Firebase's Identity Toolkit REST calls rather than hitting the live `memcab-19762` project. This is a deliberate flakiness-reduction decision:

- **Pro:** Deterministic, offline-capable, no rate limits, no leftover test accounts in a real user pool, safe for CI.
- **Con:** The mock fixtures reproduce Identity Toolkit's *observed* wire format for `firebase@^12.16.0`. They are not a documented-stable contract. A future major SDK upgrade could silently invalidate them.
- **Mitigation:** Pair this suite with a periodic (e.g. weekly) manual or scheduled smoke run against a real, non-production Firebase project to catch drift the mocks wouldn't. This is a recommendation for the team's CI setup, not something this suite does automatically.

### A.4 Exit Criteria

Regression testing before a release is considered complete when:

1. `npm test` — 100% pass, zero skipped tests without a linked tracking ticket.
2. `npm run e2e` — 100% pass on `chromium-desktop`.
3. `ng build` — zero errors, zero new warnings vs. the previous release build.
4. All **Critical** and **High** severity bugs opened during the cycle are either fixed-and-verified or explicitly deferred with sign-off (not silently left open).
5. Any new user-facing flow shipped in the release has at least one corresponding automated test (unit or E2E) — no untested happy path merges.

| TC-001 | Settings — Mobile Responsive Nav | Settings tab row does not clip or crush labels on a narrow viewport, and scrolls horizontally when its content overflows | User is authenticated; viewport set to 393×851 (iPhone 14 Pro) | 1. | TC-003 | Daily Review — TTS Playback | The speaker button on a flashcard's front face triggers Web Speech Synthesis with the word's English translation and correct language tag | User authenticated; at least one due word in an active deck; on the Daily Review flashcard screen, front face showing | 1. Observe session state (progress counter, remaining queue). | The word is NOT marked complete; it reappears later in the same session queue; session progress bar reflects the requeue, not a skip | **High** |
| TC-005 | Cross-Account Data Isolation | Decks created under one Firebase account are never visible to a second account signed in on the same browser | Two distinct Firebase test accounts available | 1. Sign in as Account A. 2. Create a uniquely-named deck. 3. Log out. 4. Sign in as Account B (same browser/session). 5. Navigate to My Decks / Game Hub / Daily Review deck pickers. | Account B sees **zero** decks belonging to Account A anywhere in the UI. Switching back to Account A restores its deck. | **Critical** |
| TC-006 | Forgot Password — Invalid Email Handling | Submitting an email Firebase rejects shows a clearly-styled (red/coral) inline error, not a silent failure or a generic browser alert | On `/#/login`, "Forgot password?" card active | 1. Enter an email address. 2. Submit "Send reset link" (with Firebase mocked/expected to return `INVALID_EMAIL`). | A `.form-error` element appears with the message "Please enter a valid email address."; element's computed text color is a red/coral tone, not the default ink color | **Medium** |
| TC-007 | Account Deletion — Race Condition Guard | Deleting an account purges that account's storage keys even though Firebase's own sign-out callback can fire before or after `deleteUser()` resolves | Authenticated test account with at least one deck | 1. Trigger account deletion (type `DELETE`, confirm). 2. Observe network/callback ordering (or, at the unit level, simulate `onAuthStateChanged(null)` firing mid-`deleteUser()`). | The correct uid's storage keys are removed regardless of callback ordering; user lands on `/login`; the deleted account can no longer sign in | **Critical** |

*Severity legend: **Critical** = data leak / auth bypass / data loss. **High** = a core user journey is broken or misleading. **Medium** = degraded UX, workaround exists. **Low** = cosmetic.*
